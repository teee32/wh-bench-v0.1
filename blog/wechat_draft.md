## 一、起因：政策给我，但我不会业务

我是一个数据科学大三在读。

最近在追武汉的政策风口，AI+OPC 一人公司、人形机器人、低空经济……每一条政策背后都跟着「最高 100 万资助」「数据集建设最高奖励 200 万」这种数字。

但我有一个尴尬的现实：**我没有项目，也讨厌跑业务。**

跑社区、约老板、谈付费意向 —— 这些话术我光想就头大。但政策文件里反复出现的那个词，让我突然看到出口：

> **「建设高质量数据集」**
>
> ——《2026 年政府工作报告》《十五五规划纲要》《武汉人形机器人三年行动方案》

数据集这条赛道的好处是：**客户找你，你不用找客户。** HuggingFace 下载量是销量，GitHub issue 是客服。全部异步、全部技术语言、全部在屏幕里。

最关键的是：政策文件里的"数据集"三个字，不是修辞，是真的写了金额。

于是我用一个周末，做了一个 v0.1 出来。

## 三、方法学的小创新：双 LLM Jury

中文 LLM 评测集这个东西，国内不是没人做。但绝大多数都做成了「我用 GPT-4 蒸馏，然后宣布质量高」。

这里有一个非常常见的盲点：**用一个模型蒸馏数据、再让同一个模型自评，会形成评分通胀。**

我的做法是：

1. **蒸馏阶段**：用 GPT-5.5-mini 从 50 部法规原文里提取 305 个问答对
2. **过滤阶段**：自动剔除空答、重复、超长 → 留下 303 条
3. **评测阶段**：用**两个不同家族**的模型背靠背审核：
   - **GPT-5.5**（评测员 A）
   - **Claude Sonnet 4.6**（评测员 B）
4. **裁决（Jury）规则**：
   - **双方都判 Fail** → 直接剔除（共 3 条）
   - **一方 Fail 一方 Pass** → 保留并标记为 `CONTESTED`，写进 dataset card 透明披露（共 15 条）
   - **双方都 Pass** → 入库（共 203 条）
   - 其他组合 → 保留（共 79 条）

最后入库 300 条，CONTESTED 15 条作为「已知 limitation」公开。

这个 **Jury 一致性矩阵**长这样：

![双 LLM Jury 一致性矩阵](https://raw.githubusercontent.com/teee32/wh-bench-v0.1/main/blog/jury_matrix.png)


```
GPT \ Claude  │ pass  borderline  fail
──────────────┼──────────────────────
pass          │ 203      72        14
borderline    │   2       8         1
fail          │   0       0         3
```

**为什么这个矩阵重要？** 因为它直接告诉用户：

- 这个数据集**不是单一模型的偏见再现**
- 我们承认有 5.9% 的样本两个 judge 看法不一致
- 你可以根据需要自己决定是用「严格 203 条」还是「保守 300 条」

国内中文评测集里，把异议公开发布的，凤毛麟角。

## 四、踩坑录：周末项目里的真实问题

不是所有事都顺利。挑三个最有代表性的坑：

### 坑 1：foxnio 的 WAF 拦了 OpenAI SDK 默认 User-Agent

调用 OpenAI 兼容 API 时直接 403，提示 "Your request was blocked"。但用 curl 同样的请求又能过。

**解决**：

```python
client = OpenAI(
    api_key=key,
    base_url=url,
    default_headers={"User-Agent": "curl/8.4.0"}
)
```

把 User-Agent 改成 curl 的，绕过 WAF。这种坑在 SDK 文档里是不会写的。

### 坑 2：蒸馏 chunking 的边界问题

最初 chunk 切分只看到了某条法规的「第一条」，但答案展开到了第三条 —— 结果蒸馏出来的 QA 对，答案在原文里找不到对应位置。

**解决**：把 chunk 范围扩展到「条文 + 后续 2 条」，并且在 system prompt 里明确写：「如果文本里没有明确答案就回复 SKIP」。

### 坑 3：Claude 比 GPT 严得多

同一份 303 条样本：
- GPT-5.5 给 fail 3 条
- Claude Sonnet 4.6 给 fail 18 条

如果只用 Claude，要剔 18 条；只用 GPT，剔 3 条。

**这不是 bug，这是 feature** —— 不同模型的"严格度"差异本身就是研究问题。所以才有了 Jury 机制：双方都说不行，才剔。

## 五、这事对我意味着什么

回到开头的问题。

我没有项目、不会跑业务、也没钱。但有一个周末和一个能调通 API 的笔记本。

做完之后我意识到，对一个**讨厌跑业务的数据科学学生**来说，「构建并开源高质量数据集」这条路有几个被严重低估的优势：

**① 政策对口，金额明确**

武汉市《人形机器人三年行动方案》原话：「对数据集的建设最高奖励 200 万元。」

不是修辞，不是「赋能」，不是「打造高地」。是**真的会发钱**，而且发钱的标准是：**数据集本身的质量和影响力**，不是你 PPT 讲得多好听。

**② 学生身份反而是优势**

数据集是体力活，但你有大把时间清洗、标注、迭代版本。这是工业界团队没有的奢侈品。学校算力、教授签字、学院盖章 —— 都比"独立开发者"多一层背书。

**③ 复利效应明显**

数据集一旦发布，**版本号是单调递增的**。v0.1 没人下，v0.3 1000 下载，v1.0 就有人发 issue 想买商业 license。**别人很难超车，因为你的版本已经在前面了。**

**④ 申报材料天然硬核**

普通申报材料要写 BP、写商业计划、写未来三年规划，这些都是"吹"。而我可以直接附上：

- HuggingFace 数据集链接（带下载量）
- GitHub repo（带 commit 记录）
- 双 LLM jury 一致性矩阵（带数据）
- DATASHEET（数据卡，工业标准）
- TECH REPORT（技术报告，论文格式）

**全部是数字。全部可以验证。全部是别人没法 fake 的。**

## 六、给同样想踩政策风口的同学

如果你也在武汉读书，也在数据科学/AI 相关专业，**这条路对你的不公平优势比创业老炮还大**。

但我必须说三句直话：

**别先注册公司再找政策。** 顺序错了直接亏几万。先把社区批文、英才计划批文拿到，再注册主体能多吃一笔"入驻奖励"或"落户奖励"。

**别做"通用 AI 助手"。** 政府文件里反复出现的关键词是「行业」「场景」「规模化应用」—— 你必须能一句话说清"我服务哪个行业的哪类人"。通用 ChatGPT wrapper 已经过剩了。

**别一个人闷头干。** 找一位本校 AI 方向老师挂名指导。教授**愿意**签字 —— 因为你的成果对学院考核也是政绩。这是一个三赢的事情，没人会拒绝。
