# 周末做了一个上线 HuggingFace 的中文法规评测集

> 一篇关于「政策风口怎么落地到代码」的实操记录

---

## 一、起因：政策给我，但我不会业务

最近在追武汉的政策风口，AI+OPC 一人公司、人形机器人、低空经济……每一条政策背后都跟着「最高 100 万资助」「数据集建设最高奖励 200 万」这种数字。

但我有一个尴尬的现实：**我没有项目，也讨厌跑业务。**

跑社区、约老板、谈付费意向 —— 这些话术我光想就头大。但政策文件里反复出现的一个词，让我突然看到出口：

> **「建设高质量数据集」**
>
> ——《2026 年政府工作报告》《十五五规划纲要》《武汉人形机器人三年行动方案》

数据集这条赛道的好处是：**客户找你，你不用找客户。** HuggingFace 下载量是销量，GitHub issue 是客服。全部异步、全部技术语言、全部在屏幕里。

最关键的是：政策文件里的「数据集」三个字后面，跟着的是**白纸黑字的金额**。

于是我用一个周末，做了一个 v0.1 出来。

## 三、方法学的小创新：双 LLM Jury

中文 LLM 评测集这个东西，国内做的人不少。但绝大多数都做成了「用 GPT-4 蒸馏，然后宣布质量高」。

这里有一个非常常见的盲点：**用一个模型蒸馏数据、再让同一个模型自评，会形成评分通胀。**

我的做法是：

1. **蒸馏阶段**：用 GPT-5.5-mini 从 50 部法规原文里提取 305 个问答对
2. **过滤阶段**：自动剔除空答、重复、超长 → 留下 303 条
3. **评测阶段**：用**两个不同家族**的模型背靠背审核：
   - **GPT-5.5**（评测员 A）
   - **Claude Sonnet 4.6**（评测员 B）
4. **裁决（Jury）规则**：
   - **双方都判 Fail** → 直接剔除（共 3 条）
   - **一方 Fail 一方 Pass** → 保留并标记为 `CONTESTED`，写进 dataset card 透明披露（共 15 条）
   - **双方都 Pass** → 入库（共 203 条）
   - 其他组合 → 保留（共 79 条）

最后入库 300 条，CONTESTED 15 条作为「已知 limitation」公开。

这个 **Jury 一致性矩阵**长这样：

![双 LLM Jury 一致性矩阵](https://raw.githubusercontent.com/teee32/wh-bench-v0.1/main/blog/jury_matrix.png)


```
GPT \ Claude  │ pass  borderline  fail
──────────────┼──────────────────────
pass          │ 203      72        14
borderline    │   2       8         1
fail          │   0       0         3
```

**为什么这个矩阵重要？** 因为它直接告诉用户：

- 数据集经过**两个家族 LLM 同时验证**，缓解了单模型自评的偏见
- 我们承认有 5.9% 的样本两个 judge 看法不一致
- 你可以根据需要自己决定是用「严格 203 条」还是「保守 300 条」

国内中文评测集里，把异议公开披露的做法，并不常见。

## 四、踩坑录：周末项目里的真实问题

挑三个最有代表性的坑：

### 坑 1：LLM 蒸馏时偷偷改写关键数值

某条电力调度规章原文里写着「负荷较预测偏差超过 **30%**，需启动应急减供」。蒸馏出来的 QA 对里，模型把 30% 写成了 50%。

原文一字没漏，但 LLM 在生成"答案"的时候自作主张"修正"了一个它觉得"看起来不对"的数字。

**解决**：在 LLM judge 阶段加了一个「数值锚定」模板 —— 强制评测员把答案里出现的所有数字与原文逐一对齐。最终被剔除的第 0176 条就是这么被两个 judge 同时抓到的。

**教训**：法规、金额、日期、百分比，必须有独立校验环节。LLM 在这种地方的"幻觉"非常隐蔽，肉眼一遍读过去根本看不出来。

### 坑 2：蒸馏 chunking 的边界问题

最初 chunk 切分只看到了某条法规的「第一条」，但答案展开到了第三条 —— 结果蒸馏出来的 QA 对，答案在原文里找不到对应位置。

**解决**：把 chunk 范围扩展到「条文 + 后续 2 条」，并且在 system prompt 里明确写：「如果文本里没有明确答案就回复 SKIP」。

### 坑 3：Claude 比 GPT 严得多

同一份 303 条样本：
- GPT-5.5 给 fail 3 条
- Claude Sonnet 4.6 给 fail 18 条

如果只用 Claude，要剔 18 条；只用 GPT，剔 3 条。

**这本身就是一个值得展示的差异现象** —— 不同模型对同一份数据的"严格度"差异，正是评测领域里一直缺少正面讨论的话题。Jury 机制就是为了把这种差异显式地呈现出来：双方都说不行，才剔。

## 五、这事对我意味着什么

回到开头的问题。

我没有项目、不会跑业务、也没钱。但有一个周末和一个能调通 API 的笔记本。

做完之后我意识到，对一个**讨厌跑业务的人**来说，「构建并开源数据集」这条路有几个被严重低估的优势：

**① 政策对口，金额明确**

武汉市《人形机器人三年行动方案》原话：「对数据集的建设最高奖励 200 万元。」

白纸黑字，金额具体，预算有据可查。评审的核心抓手是**数据集本身的质量和影响力**，PPT 讲得再花也代替不了实测数据 —— 这对不擅长讲故事的技术人是一个相对友好的赛道。

**② 时间和体力是被严重低估的资源**

数据集是个体力活，需要大把时间做清洗、标注、迭代版本 —— 这恰恰是有 KPI 压力的工业界团队反而抽不出手的事情，而是非全职开发者、在校学生、转行者最有优势的区间。如果你还能挂上一位领域内老师指导，那就是技术成果 + 学院背书的双重叠加。

**③ 复利效应明显**

数据集一旦发布，**版本号是单调递增的**。v0.1 没人下不要紧 —— 只要你持续迭代到 v0.3、v1.0，前期累积的 commit 历史、issue 反馈、下载曲线，都会变成新入场的人短期内复制不出来的资产。**这是时间换来的，没法用钱跳过。**

**④ 申报材料天然硬核**

普通申报材料要写 BP、写商业计划、写未来三年规划，这些内容里"吹"的成分谁都心知肚明。而我可以直接附上：

- HuggingFace 数据集链接（带下载量）
- GitHub repo（带 commit 记录）
- 双 LLM jury 一致性矩阵（带数据）
- DATASHEET（数据卡，参考 Gebru 等人提出的业界推荐格式）
- TECH REPORT（技术报告，含完整方法学披露）

**全部是数字。全部可以查证。全部不是 PPT 能堆出来的。**

## 六、给同样想踩政策风口的人

如果你也在武汉，也在做 AI/数据相关的事，**这条路给你的不公平优势比创业老炮还大**。

但我必须说三句直话：

**别先注册公司再找政策。** 顺序错了直接亏几万。先把社区批文、英才计划批文拿到，再注册主体能多吃一笔"入驻奖励"或"落户奖励"。

**别做"通用 AI 助手"。** 政府文件里反复出现的关键词是「行业」「场景」「规模化应用」—— 你必须能一句话说清"我服务哪个行业的哪类人"。通用 ChatGPT wrapper 已经过剩了。

**别一个人闷头干。** 找一位熟悉领域的老师挂名指导。很多教授其实**愿意**签字 —— 因为你的成果对学院考核也算政绩。只要项目本身能讲清楚，被拒绝的概率比你想象的低很多。
